# DECISIONS — Nexus Brain

> `/arch-draft` ADR log (Brain-1 rough draft). Each entry: decision · status · rationale · alternatives · what-would-change-it. Statuses: `locked` (owner-confirmed) · `proposed` (recommended, not yet confirmed) · `open`. Mode: Expanded · Posture: Production-grade. Dates 2026-06-15/16.

## D-1 — Planning mode Expanded, posture Production-grade `[locked]`
Expanded artifact set (incl. `THREAT_MODEL.md` + `DATA_MODEL.md`); production-grade quality bar on load-bearing paths even though scope is the standalone MVP. *Alt:* Standard/MVP-posture — declined (privacy/secret surface + trust north star justify the depth). *Changes if:* the build is re-scoped to a throwaway prototype.

## D-2 — Federation from day one `[locked — owner]`
The MVP ships the federation router + multi-project registry + N-store union-read + cross-repo query. **Cross-repo *symbol/edge* resolution is `research required`** with a graceful fallback ("N graphs side-by-side", per-project answers union-ranked, marked). *Alt:* single-project-first then federate — declined (owner wants the portfolio differentiator up front). *Changes if:* the federation spike shows union-read latency/quality is unacceptable → fall back to single-project-first.

## D-3 — Two surfaces, embedded agent is the MVP face `[locked — owner]`
An **embedded frontier-model agent** (Claude API) behind a standalone chat UI + CLI, **and** an **MCP server** for external agents (Claude Code/Codex/CI). Both are clients of **one internal retrieval core**; the MCP server is the shared trust boundary. *Alt:* MCP-server-first / own-agent-only — declined.

## D-4 — Standalone mutation = bounded allowlist; integrated = propose-only `[locked — owner]`
Standalone, the agent may directly (with preview + consent): write its own `~/.project-brain/` store + manifests; **owned-doc refresh** with don't-clobber/3-way-merge; idempotent/reversible/consented host-config during `setup`. Else propose-only/out-of-scope. **Integrated → strictly propose-only via the NexusOps Gateway.** *Alt:* strictly-propose-only everywhere / full local autonomy — declined.

## D-5 — Trust posture: single-user local + optional loopback HTTP `[locked — owner]`
One OS user, one machine. Primary transport = **stdio MCP** (no port). Optional opt-in **loopback HTTP** (`127.0.0.1` + per-launch token) for a separate-process/browser UI. Secrets in **OS keychain**; per-project `policy.yaml` + Redactor enforced at the MCP boundary regardless of caller. No multi-user RBAC. *Changes if:* a remote/shared mode is ever in scope (currently a non-goal).

## D-6 — North star = trust / citation correctness `[locked — owner]`
Every served claim carries a **live, re-validated `file:line` anchor**; refuse-or-flag the unsupported; a confident-but-wrong or stale citation is THE cardinal failure. Freshness, privacy, federation are load-bearing supports beneath this bar.

## D-7 — Grounding gate = answer-but-flag (+ strict mode) `[locked — owner]`
Never present an ungrounded claim as cited; **post-validate every cited span exists**; separate/mark "couldn't ground: X" at low confidence; opt-in **strict mode** refuses under insufficient grounding. Backed by the 3-layer grounding stack (Contextual Retrieval at index time · native Citations API · post-validation contract).

## D-8 — Orchestration = LlamaIndex (Workflows / AgentWorkflow) `[locked — owner]`
DIY-but-structured agent loop on **LlamaIndex** (parsing + retriever assembly + event-driven Workflows); **no LangChain / LangGraph**. *Alt:* raw Anthropic SDK (lighter), LangGraph (heavier; durable execution maps weakly because Nexus Brain is propose-only / the heavy execution is NexusOps's). *Changes if:* a surface needs durable/long-running resumable stateful execution → reconsider LangGraph.

## D-9 — Observability = OTel-first, layered, Collector-fanned-out `[locked — owner]`
**Instrument once** with OpenTelemetry + **OpenInference** (auto-instruments LlamaIndex + Anthropic → standard OTel). An **OpenTelemetry Collector** is the hub and **fans out**: `gen_ai.*` spans → **Langfuse** (layer-2: LLM traces, evals, datasets, prompt mgmt); **all** spans + metrics + logs → **SigNoz** (layer-1: full operational APM — traces/metrics/logs/alerts; **Grafana LGTM** the noted alt). A thin internal **`ObservabilitySink` seam** keeps the backend swappable (config, not code). **Backends are dev/CI/ops infra, not an end-user runtime dependency** (the product only emits OTel). *Why both:* Langfuse deliberately *filters out* non-LLM spans and has no infra metrics/logs/alerting → it cannot be layer-1. *Alt:* LangSmith (cloud-default + privacy tension) — declined; Phoenix (lighter, strong span-level evals) — viable swap-in via OTel.

## D-10 — Evals = bespoke CI-gated harness + Langfuse `[locked — owner]`
A first-class, **CI-gated** eval harness (cc-crew `/eval` path) with **golden datasets versioned in-repo** and **custom evaluators** for the bespoke trust metrics: citation precision/recall, **grounding-gate correctness**, anchor-revalidation accuracy, retrieval **Recall@k**, **redaction-recall fuzz** (zero-leak security eval), freshness/drift latency, federation correctness. Scores/datasets/experiments live in **Langfuse**. The harness is load-bearing (it defends D-6); the UI is swappable.

## D-11 — Vector DB = LanceDB, one dataset per project `[locked — owner]`
**LanceDB** (embedded, columnar, dense + native BM25 + hybrid in one table, `merge_insert` upsert, SQL metadata filters, larger-than-memory, **native versioning/time-travel**). **One dataset per project** (PRD `PB-1`: per-project stores, not one giant index) → isolation + independent idle-eviction + per-project blue-green re-embed. Design around the `optimize()`-after-write fold requirement. *Alt:* sqlite-vec (lighter, < ~1M chunks) — viable fallback; one-big-table-with-`project`-filter — declined (loses per-project lifecycle).

## D-12 — Three-store model, fused at query time `[locked]`
**① LanceDB** (embeddings + BM25 + anchors; project-brain-owned) · **② CodeGraph SQLite** (`.codegraph/`, structural symbol graph; external tool owns it, Nexus Brain reads it) · **③ `.project-brain/manifest.json` + `~/.project-brain/` registry** (catalog: hashes/versions/SHA/freshness). **Not merged** — the agent fuses semantic (①) + structural (②) at query time. CodeGraph is a `setup`-installed external dependency; Nexus Brain never writes it.

## D-13 — RAG pipeline `[locked]`
Hybrid (dense + BM25) → rerank (~30–50 → ~10) → **whole-file hydration** → frontier model (Contextual Retrieval + Citations API) → grounding gate. **Routing:** per-project corpus < ~200K tokens → cached **long-context** (skip retrieval); larger/many → **agentic RAG** (LlamaIndex Workflow). **Graph tools always-on.** Chunking: docs heading-split + late-chunking; code AST (`CodeHierarchyNodeParser`) + Contextual Retrieval blurbs.

## D-14 — Versioning & freshness ("nothing stale silently") `[locked — owner ask]`
(1) **LanceDB native versioning** — every write a new version; `list_versions()`/`restore()`/**tags**; **tag each ingest with the repo git SHA**. (2) **Per-chunk provenance** (`source_path`, `file:line`, `content_hash`, `last_resolved_sha`, `embedding_model_version`). (3) **Reproducibility invariant:** source + model-version-stamp deterministically reproduce the index (a derived cache; source is the only non-reproducible asset). (4) **Drift radar + git-hooks** (`post-commit/merge/checkout`) → diff content-hashes → mark `live|stale|moved|unknown`; hooks guarantee convergence. (5) **Blue-green re-embed** on model/dim change (new generation + atomic swap + rollback; never in-place).

## D-15 — Redaction-before-embed + secrets-in-keychain `[locked]`
The Redactor masks/quarantines secrets/PII **before embed and before any cloud-bound generation** (embedding-inversion threat); raw transcripts never embedded; `thinking` blocks excluded; secrets only as keychain refs, never in the index/config. The SAME redactor gates the MCP-boundary egress. Detailed in `THREAT_MODEL.md`.

## D-16 — Platform scope: macOS-first, Linux-ready `[locked — owner]`
Validate **macOS** in the MVP (like NexusOps); build to the OS-service seam so **Linux** (`systemd --user`) is a designed-in follow-on, not a rewrite (launchd ∥ systemd behind one interface). Windows out of scope. *Changes if:* a Linux user becomes primary sooner.

## D-17 — Standalone UI = native desktop app (Tauri) + bundled Python core sidecar `[locked — owner]`
**No web app.** The standalone product is an **actual desktop app** built on **Tauri** (Rust host + system WebView), reusing the `NexusOps-ui-kit` design system and aligning with NexusOps (also Tauri) for the eventual drawer. It **bundles the Python core** (engine + MCP server + embedded agent) **as a sidecar** (PyInstaller, daemon-owned lifecycle — exactly NexusOps's sidecar pattern), talking over **loopback HTTP + per-launch token** (D-5) or stdio. *Alt:* Electron (heavier), native SwiftUI (macOS-locked, no ui-kit reuse), TUI (minimalist) — declined. *Architecture implication:* two-process desktop app (Tauri shell ↔ Python core); the core is independently runnable headless (CLI/MCP) without the UI.

## D-18 — No timebox; correctness over speed `[locked — owner]`
"Takes as long as it takes, as long as the foundation is **production-grade** and architecturally correct/best-practice." Reinforces D-1. **Recommended first vertical slice** (sequencing, not a deadline): `add` one repo → ingest (chunk·redact·embed·stamp) → hybrid + graph retrieve → grounded answer with live anchors → **eval harness green** — proves the north star end-to-end on one project before federation breadth.

## D-19 — Product name = **Nexus Brain** (short: **Nexus**) `[locked — owner]`
The standalone product is **Nexus Brain**. Naming coherence with the platform: **NexusOps** = the control plane; **Nexus Brain** = its memory/reasoning engine. Repo dir stays `Nexus Brain` for now (dir name ≠ product name); the integration-seam actor remains `project_brain` / "Nexus Brain" per `MAIN_PLATFORM_INTERFACE.md`. Docs to be threaded to "Nexus Brain" (rename pass at draft end).

## D-20 — Distribution: signed `.dmg` + Homebrew Cask + bundled sidecar; deps provisioned by `setup` `[locked — owner gate 2026-06-16]`
Standalone **Nexus Brain** ships as a **signed + notarized macOS `.dmg`/`.app`** (Tauri, with the auto-updater) and a **Homebrew Cask** (`brew install --cask nexus-brain`). The **CLI** (`nexus`/`nb`) ships inside the bundle (PATH-symlinked at setup) and is also installable headless via a **Homebrew formula / pipx** for servers/CI. The app **bundles the Python core sidecar** (PyInstaller — no system Python needed). External runtime deps (**CodeGraph**, **Ollama + the local embed model**) are **provisioned by `setup`/`add`** (detect→install→verify, idempotent + consented + progress UX) rather than bloating the installer. **npm is NOT the primary** channel (the core is Python; npm would be a thin shell only) — supersedes the ONBOARDING doc's `npm i -g` idea. Linux later (D-16): AppImage/tarball + systemd. *Changes if:* a multi-GB model must ship offline (then a separate "offline bundle").

## D-21 — One core, two host adapters (ports-and-adapters); monorepo + published core `[locked — owner gate 2026-06-16]`
The **core engine** (ingest · index · retrieval · grounding · drift · federation · agent loop) is **delivery- and host-agnostic**, talking to the world only through **ports**: `HostPort` (authorize + perform side-effects), `EventSource` (git/watcher vs the NexusOps event outbox), `EmbeddingProvider`/`ModelProvider` (pluggable, O-A), `ObservabilitySink` (D-9), `SecretStore` (keychain). **Two adapters implement `HostPort`:** **StandaloneHost** (the D-4 bounded-mutation allowlist + embedded Claude agent + desktop/CLI/MCP surfaces, reads git directly) and **NexusOpsHost** (LATER — propose-only: every mutation → a typed `ActionPlan`/`ActionRequest` to the Gateway; consumes the platform outbox; shared IDs; the drawer surface — exactly `MAIN_PLATFORM_INTERFACE.md` v0.2). **The core never forks**; embedded = the same engine + the NexusOpsHost adapter (additive at the edge). **Build the `HostPort` seam from day one** (D-4 already forces the distinction) so the embedded adapter slots in with zero core refactor. **Repo:** one **monorepo** — `nexus-brain-core` (pip-publishable) + `-cli`/`-mcp` entrypoints + `-app` (Tauri standalone) + later `-nexusops` (the adapter). NexusOps consumes the **published core package** (versioned dep), never a fork; split to a 2nd repo only if/when NexusOps needs an independent release cadence. The standalone↔embedded contract = `MAIN_PLATFORM_INTERFACE.md` + the core's public API.

## D-22 — Ship instrumented-but-silent; obs backends + evals are dev/CI only; never phone home `[locked — owner gate 2026-06-16]`
**Three things, three answers:** (1) **Eval harness + golden datasets + Langfuse/SigNoz backends = developer/CI infrastructure — NOT shipped** (they live in the repo + CI). (2) **OTel instrumentation = shipped, but OFF by default, local-only, opt-in** ("diagnostics / local tracing" toggle that can point only at the user's OWN local collector → cheap no-op when unwired; lets a user capture a local trace for a bug report). (3) **Telemetry to the developer = NEVER** — a privacy-first local tool must not exfiltrate usage/telemetry (no phone-home, ever — the same reason LangSmith's beacon was a dealbreaker). The `ObservabilitySink` seam (D-9) means the instrumentation is always present but unwired by default.

## D-23 — Embedding + rerank providers (verified SOTA, 2026-06-16) `[locked — research-backed]`
Resolves O-A. **Pluggable, version-stamped providers** chosen per-project in `policy.yaml` at `add`/setup (blue-green re-embed on change, D-14). Verified-current defaults (a 9-agent adversarial research pass, 2026-06-16, upgraded the local picks vs the 06-03 dossier):
- **Local embedding default = `qwen3-embedding-4b`** (Apache-2.0, ~2.5 GB Ollama, 40K ctx, Matryoshka dims; dense-only → pair with the BM25 leg). *Upgrade from bge-m3* (~80.06 vs no code-specific score). Fallbacks: `qwen3-embedding-0.6b` (639 MB), `bge-m3` (MIT; single-model dense+sparse+ColBERT), `embeddinggemma` (thin-laptop, ~2K ctx).
- **Cloud embedding opt-in default = `voyage-code-3`** (confirmed; 32K ctx, Matryoshka+quant, ~13–17% code lead; no voyage-code-4 exists). Registered alt: `gemini-embedding-2` (MTEB-Code 84.0 but vendor-only-vs-Voyage → bake-off before promoting). Privacy fallback: `openai-text-embedding-3-large`.
- **Local reranker = `qwen3-reranker`** (0.6b laptop / 4b GPU, Apache-2.0; MTEB-Code 81.20 vs bge-reranker-v2-m3 41.38 — *upgrade*). Alts: `mxbai-rerank-large-v2`; bge-reranker demoted to fallback.
- **Cloud reranker default = `voyage-rerank-2.5`** (stack-coherent), but **not the leader** → run an in-domain bake-off vs `zerank-2` (#1 ELO, ~50% cheaper) + `cohere-rerank-4-pro` (#2).
- **Context augmentation:** cloud = `voyage-context-3` (~6.76% chunk gain, no per-chunk LLM call); local = Anthropic Contextual Retrieval / late-chunking.
- **Privacy = explicit user choice, `local | cloud`, per project `[locked — owner gate 2026-06-16]`:** at `add`/setup the user picks **local** (default — on-device, nothing leaves the machine) or **cloud** (best quality) for embedding + generation, surfaced with an **honest one-line disclosure** of the cloud tradeoff (your code is sent to the provider; standard plans retain it ~30 days). **No ZDR-enforced / no "opt-out enforced" as HARD invariants** — both are un-enforceable account-level settings (ZDR = a separately-negotiated enterprise contract; Voyage opt-out = an org-dashboard toggle) → we **disclose**, we don't pretend to gate. `ModelProvider` defaults to the **latest capable Claude** (Opus 4.8 / Sonnet 4.6), pluggable + ZDR-aware (pins a ZDR-eligible model only *if* a ZDR profile is configured; no Fable assumption). **Redaction-before-embed + keychain-only secrets apply regardless of local/cloud** (those stay hard invariants). *(Resolves audit C-9/C-10/LBD-9/LBD-10 by simplification.)*

## D-25 — LanceDB CONFIRMED (adversarial verification, 2026-06-16) + maintenance contract `[locked — research-backed]`
A 10-agent adversarial pass (steelmanned the rivals, hunted LanceDB's weaknesses) confirms LanceDB **decisively**: it is the **only embedded/local-first store covering R1 (in-process, no server) + R3 (git-SHA-taggable versioning) + R7 (larger-than-memory) together.** Closest rivals each fail a hard requirement — Chroma (no versioning + macOS RAM blow-ups), **sqlite-vec** (pre-v1 storage-format churn + no versioning + brute-force, ~280× slower at 100k), DuckDB-VSS (crash-unsafe HNSW persistence → data loss on unclean quit), Qdrant-local/Milvus-Lite (dev-only), Weaviate-embedded (spawns a server, not in-process). **Confirm with eyes open:** the high-level `lancedb` API is pre-1.0 Alpha on a **stable substrate** (Lance SDK 1.0 / file format 2.1) → pin it. Apache-2.0 + open portable Lance format = low lock-in (data stays readable, incl. via DuckDB, if the vendor pivots).
**Maintenance contract (FIRST-CLASS, day-one deliverable):** (1) `optimize()` after each upsert batch + monitor `index_stats().num_unindexed_rows ≈ 0` (W1: post-write rows fall to flat scan until folded; `fast_search` silently excludes them); (2) scheduled `cleanup_old_versions()` + **tag every build with its git SHA** (tagged versions are GC-exempt → doubles as R3) (W2: versions accrue → TB bloat); (3) **single-writer-per-dataset** + **read-only federation fan-out** (W3: optimistic-commit conflicts — this is *why* the read-only router is correct); (4) RAM-bounded batched index builds; per-project datasets keep each small (W4); (5) pin `lancedb`, version-appropriate handle pattern, periodically recycle the sidecar (W5); use **`spawn` not `fork`**, verify **arm64 wheels** in CI.
**Federation (R6, refines D-2):** build the fan-out + RRF rank-fusion router; **optionally back it with the DuckDB Lance core extension** (May 2026 — ATTACH + JOIN + hybrid over Lance datasets **in-process**, keeps R1). **FTS caveat:** native Lance FTS has no boolean AND/OR in the query string → model boolean logic in the router; enable `with_position` only where phrase queries are needed; combine with metadata pre-filters for exact-identifier scoping.
**Flip conditions (revisitable lock, not faith):** R3 ever dropped → sqlite-vec becomes competitive · a rival ships R1+R3+R7 under a permissive license · `lancedb` never reaches 1.0 / churn worsens · OSS feature stratification (watch the Lakehouse roadmap) · the maintenance contract proves unbearable on end-user hardware.
