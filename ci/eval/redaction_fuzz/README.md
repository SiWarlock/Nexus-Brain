# Redaction Fuzz Harness — Spike 0.1

**Purpose:** De-risks the Phase-1.5 `Redactor` interface freeze and Phase-2.3 engine build.
**Anchors:** `ARCHITECTURE.md §18` · `THREAT_MODEL.md §Redaction engine` · `DECISIONS.md D-15/D-26/C-11`

## What this is

A reusable measurement rig that takes any `redact(payload, sink) -> str` callable and reports:
- **Recall** over the catchable set (§18 classes)
- **False-positive rate** over non-secret look-alikes
- Whether the engine passes the **proposed envelope** (`PROPOSED_RECALL_FLOOR=0.95`, `PROPOSED_FP_CEILING=0.05`)

The real redaction engine lands in Phase-2.3 (`core/redactor/`). This harness is the CI gate it must pass.

## Module layout

```
ci/eval/redaction_fuzz/
  __init__.py          — package + safety contract header
  types.py             — Sink enum, SecretClass enum, AcceptedResidualClass enum,
                         SecretSample/NonSecretSample, RedactionResult/FPResult,
                         HarnessReport, RedactorProtocol
  generator.py         — Property generator: synthetic secret-shaped inputs (all §18 classes)
                         + non-secret look-alikes for FP measurement
  corpus.py            — Curated adversarial seed corpus: split secrets, base64, comment/URL
                         embedding, env dumps, accepted-residual edge cases
  oracle.py            — Leak oracle: check_leak(), check_false_positive(), is_gate_failure()
  harness.py           — run_harness(redact_fn, sinks, verbose) -> dict[Sink, HarnessReport]
                         + proposed envelope constants + CLI entry point
  stub_redactor.py     — ⚠ REFERENCE STUB ONLY — not the real engine; validates harness wiring
  test_harness_infra.py — pytest suite for harness infrastructure (34 tests, all green)
```

## Usage

### As a pytest suite (harness validation)

```bash
cd project-brain-contract
# Requires uv env from core/ — run from there:
cd core && uv run python3 -m pytest ../ci/eval/redaction_fuzz/test_harness_infra.py -v
```

### As a CLI tool (stub baseline)

```bash
cd project-brain-contract
python3 -m ci.eval.redaction_fuzz.harness
python3 -m ci.eval.redaction_fuzz.harness --verbose
python3 -m ci.eval.redaction_fuzz.harness --sink cloud_egress --verbose
```

### Plugging in the real Phase-2.3 engine

```python
from ci.eval.redaction_fuzz.harness import run_harness, PROPOSED_RECALL_FLOOR, PROPOSED_FP_CEILING
from core.redactor import redact  # Phase-2.3

reports = run_harness(redact)
for sink, report in reports.items():
    assert report.gate_pass(PROPOSED_RECALL_FLOOR, PROPOSED_FP_CEILING), (
        f"Redactor failed gate on {sink}: recall={report.recall:.1%} fp={report.fp_rate:.1%}"
    )
```

## Stub baseline (harness validation — NOT the Phase-2.3 target)

The stub deliberately fails the gate to prove the harness is non-trivial:

| Sink | Recall | FP rate | Gate |
|---|---|---|---|
| persist | 77.6% | 0.0% | FAIL (recall < 95%) |
| mcp_egress | 77.6% | 0.0% | FAIL (recall < 95%) |
| cloud_egress | 77.6% | 0.0% | FAIL (recall < 95%) |

What the stub misses (by design): high-entropy KV without a sensitive key name, base64-encoded secrets, JWT tokens (not prefixed in the right context), secrets in URL query params.

## Corpus statistics

- **Generated samples (property-based):** ~150 across 4 classes
- **Adversarial corpus (curated):** 20 samples (split secrets, comments, URLs, base64, env dumps, accepted-residual edges)
- **Non-secret look-alikes:** 26 samples (git SHAs, UUIDs, code literals, short JSON values)
- **Catchable-set samples per run:** 125 (residuals excluded from the recall gate)
- **Accepted residuals per run:** 8 (expected to pass through; not gate failures)

## Safety invariants

- **Synthetic secrets only.** All plaintext values are generated by a deterministic seeded RNG — they match the shape of real secrets but are not real credentials.
- **Zero network egress.** MCP-egress and cloud-egress sinks are simulated in-memory string operations. No HTTP calls are made.
- **Three-sink coverage.** Every run exercises all three sinks: `persist`, `mcp_egress`, `cloud_egress`.

## What Phase-2.3 must do

1. Replace `stub_redact` with `core.redactor.redact` and pass the harness.
2. Achieve `recall >= 0.95` on the catchable set across all three sinks.
3. Achieve `fp_rate <= 0.05` on non-secret look-alikes.
4. Achieve `fp_rate == 0.0` on git SHA samples specifically (hard sub-invariant; LanceDB version tagging).
5. Be idempotent and never raise.
6. Add this harness invocation to the CI pipeline as a hard gate.

See `docs/audits/redaction-envelope.md` for the full envelope specification.
